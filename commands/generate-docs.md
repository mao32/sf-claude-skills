# /generate-docs — Genera documentazione tecnica e funzionale

Genera la documentazione completa per una user story in formato .docx
(importabile direttamente in Google Docs).

## Input

$ARGUMENTS deve contenere l'ID della user story (es. US-1234).
Se non fornito, chiedi quale user story documentare.

## Istruzioni

Genera **due documenti** nella cartella `docs/`:

---

### Documento 1: Documentazione Tecnica

**File**: `docs/TECH_<US-ID>.docx`
**Destinatario**: Architetto, tech lead, sviluppatori

Usa la docx skill (npm docx-js) per creare un documento .docx professionale
con le seguenti sezioni. Formato A4, font Arial, margini 1 inch.

**Struttura del documento:**

1. **Intestazione** (Header)
   - Titolo: "Technical Design Document"
   - Sottotitolo: US-ID + titolo breve della user story
   - Data di generazione
   - Autore: "Generated by Claude Code"

2. **Overview**
   - Riepilogo della user story (2-3 frasi)
   - Criteri di accettazione (lista numerata)

3. **Data Model**
   - Tabella degli oggetti coinvolti (Nome, Tipo, Azione: NEW/MODIFIED/EXISTING)
   - Tabella dei campi per ogni oggetto nuovo/modificato (API Name, Type, Length, Required, Description)
   - ERD in formato testuale o descrizione delle relazioni

4. **Architecture & Design**
   - Pattern utilizzati (Trigger Handler, Service Layer, Selector, ecc.)
   - Diagramma di flusso della logica (descritto passo per passo)
   - Classi Apex create/modificate con responsabilità di ciascuna
   - Tabella: Classe → Tipo → Metodi principali → Descrizione

5. **Automation**
   - Flow creati/modificati (nome, tipo, trigger, descrizione)
   - Trigger e handler (oggetto, eventi, logica)
   - Validation Rule (oggetto, condizione, messaggio errore)

6. **Security**
   - Permission Set creati (nome, permessi inclusi)
   - Considerazioni su sharing rules e field-level security
   - Profili impattati

7. **Test Strategy**
   - Tabella scenari di test:
     | # | Scenario | Tipo | Input | Risultato atteso | Classe test |
   - Coverage target e risultati effettivi (se disponibili post-deploy)
   - Dati di test utilizzati (descrizione del TestDataFactory)

8. **Deployment**
   - Lista metadata nel package.xml (raggruppati per tipo)
   - Dipendenze e ordine di deploy
   - Istruzioni post-deploy (se necessarie)

9. **Rischi e Note**
   - Rischi identificati e mitigazioni
   - Impatti su funzionalità esistenti
   - Note per la review dell'architetto

---

### Documento 2: Documentazione Funzionale

**File**: `docs/FUNC_<US-ID>.docx`
**Destinatario**: Business analyst, product owner, utenti finali

Stessa formattazione docx professionale.

**Struttura del documento:**

1. **Intestazione**
   - Titolo: "Documentazione Funzionale"
   - Sottotitolo: US-ID + titolo breve
   - Data, versione

2. **Contesto e obiettivo**
   - Descrizione del bisogno di business (in linguaggio non tecnico)
   - Benefici attesi
   - Utenti coinvolti (ruoli/profili)

3. **Descrizione funzionale**
   - Cosa succede passo per passo (dal punto di vista dell'utente)
   - Condizioni e regole di business in linguaggio chiaro
   - Esempi concreti: "Quando Mario chiude l'Opportunity X, il sistema crea..."

4. **Impatto sulle operazioni quotidiane**
   - Cosa cambia per gli utenti rispetto a prima
   - Nuovi campi/oggetti visibili e dove trovarli
   - Nuove automazioni attive

5. **Regole di validazione**
   - Lista delle validazioni con messaggio utente
   - Quando si attivano e come risolverle

6. **Permessi e accessi**
   - Chi può vedere/modificare i nuovi dati
   - Come richiedere l'accesso (se necessario)

7. **Scenari di test funzionali**
   - Tabella con scenari di verifica per l'utente:
     | Passo | Azione utente | Risultato atteso |
   - Scenari positivi e negativi in linguaggio semplice

8. **FAQ**
   - Domande frequenti anticipate con risposte

## Regole per la generazione docx

- Usare `npm install -g docx` (docx-js) per generare i file
- Formato A4, font Arial 12pt, margini 1 inch
- Titoli: Heading 1 (16pt bold), Heading 2 (14pt bold)
- Tabelle con header colorato (sfondo #D5E8F0), bordi #CCCCCC
- SEMPRE usare WidthType.DXA per le tabelle (mai percentuali — si rompono in Google Docs)
- Mai usare \n — usare Paragraph separati
- Mai usare bullet unicode — usare LevelFormat.BULLET
- Copiare i file finali in /mnt/user-data/outputs/ se in ambiente Claude.ai

## Validazione

Dopo aver generato ciascun .docx:

1. Verifica che il file esista e abbia dimensione > 0
2. Verifica che sia un file ZIP valido (i .docx sono archivi ZIP):
   ```bash
   unzip -t docs/TECH_<US-ID>.docx
   unzip -t docs/FUNC_<US-ID>.docx
   ```
3. Verifica che contenga il documento XML principale:
   ```bash
   unzip -l docs/TECH_<US-ID>.docx | grep "word/document.xml"
   ```
4. Se uno dei check fallisce, rigenera il file correggendo il problema

## Output

Dopo la generazione, mostra:
- Conferma dei file creati
- Riepilogo del contenuto di ciascun documento
- Istruzione: "Carica i file .docx su Google Drive — si apriranno direttamente come Google Docs"
